sdd-govplan v2.0

Governance Planning para Spec-Driven Development — la capa de gobernabilidad que captura contexto, features, arquitectura y criterios de aceptación antes del SDD pipeline.
★ Flujo completo
sdd-init → sdd-govplan → sdd-propose → sdd-spec → sdd-design → sdd-tasks → sdd-apply → sdd-verify → sdd-archive
⚙ Modos de brief — LITE / STANDARD / FULL

Para evitar sobreingeniería, sdd-govplan se adapta al proyecto. La auto-detección resuelve el modo según las primeras respuestas, o podés forzarlo con flags:

Modo Secciones Preguntas Ideal para Flag
LITE 7 ~6-8 CLI tools, MVPs, protos, APIs simples --lite / -l
STANDARD 11 ~12-15 Web apps, equipos 2-5, proyectos medianos --standard / -s
FULL 19 ~20-25 Sistemas críticos, equipos 6+, high/critical --full / -f

Auto-detección: cli_tool + 1 persona + mvp-2sem → LITE • web_saas + 2-5 + medium-3mes → STANDARD • 6+ / critical / long-6mes+ → FULL
El usuario puede cambiar de modo en cualquier momento diciendo "cambiar a lite" o "cambiar a full".

❓ Preguntas frecuentes
¿Por cuál empiezo: sdd-govplan o sdd-init?
Da igual. El orchestrator tiene un SDD Init Guard que, cuando ejecutás /sdd-govplan, verifica automáticamente si el proyecto ya tiene init. Si no lo tiene, lo corre silenciosamente primero y después arranca con el governance planning. No tenés que pensar en el orden.
¿Qué diferencia hay entre sdd-init y sdd-govplan?
sdd-init es infraestructura: detecta el stack del proyecto, las herramientas de testing, configura la persistencia (engram / openspec).
sdd-govplan es contenido: te hace preguntas estructuradas para entender qué vas a construir y por qué, y genera el project brief con features, decisiones de arquitectura, criterios de aceptación y tareas de alto nivel.

Uno prepara el terreno, el otro define el rumbo.
¿Skill o agente? ¿Por qué skill?
Skill. Porque:
  • Integra naturalmente con el flujo del orchestrator (via skill() tool)
  • El orchestrator maneja la conversación él mismo (preguntar no infla el contexto)
  • Delega investigación pesada a sub-agents existentes (sdd-explore)
  • No necesita un nuevo sub-agent type ni registro adicional en opencode.json
  • Sigue el patrón de las demás skills SDD (mismo ORCHESTRATOR GATE, mismo contrato de persistencia)
Un agente nuevo requeriría registrarlo en opencode.json con modelo, tools y permisos. Al pedo si el orchestrator puede hacer el trabajo pesado de coordinación.

El template completo tiene 35 secciones organizadas en: metadata, negocio, stakeholders, arquitectura, datos, file storage, search, frontend, auth, API, agentes AI, observabilidad, CI/CD, testing, seguridad, features registry, ADRs, requisitos técnicos y riesgos.
¿Cómo se decide si usar LITE, STANDARD o FULL?
Se auto-detecta según las primeras respuestas. Si ponés cli_tool + team_size: 1 + timeframe: mvp-2sem automáticamente entra en LITE (~6 preguntas, 5 minutos).

Si el proyecto es un web_saas + critical + 15+ salta a FULL (19 secciones completas).

También podés forzar con flags: /sdd-govplan --lite, /sdd-govplan --standard, /sdd-govplan --full. O cambiás de modo durante la conversación diciendo "cambiar a full".
¿Qué pasa si el proyecto es greenfield (no hay código)?
Se saltea la Fase 2 (Codebase Exploration). El governance planning se basa únicamente en las respuestas de la conversación. Para proyectos brownfield o migración, la exploración del codebase es obligatoria y se delega a sdd-explore.
¿Cómo se conecta con el resto del pipeline SDD?
Cada feature identificada en el governance brief se convierte en un SDD change. El flujo es: 
govplan → sdd-propose → sdd-spec → sdd-design → sdd-tasks → sdd-apply → sdd-verify → sdd-archive
Las architecture decisions alimentan sdd-design como ADRs. Los acceptance criteria alimentan sdd-verify como checkpoints. Los technical requirements alimentan sdd-spec como escenarios.
¿Dónde está instalado?
En dos lugares para cobertura global:
  • ~/.config/opencode/skills/sdd-govplan/ — skills de OpenCode
  • ~/.agents/skills/sdd-govplan/ — skills globales del ecosistema Gentle AI
El orchestrator lo encuentra via available_skills porque está registrado en AGENTS.md y el prompt en opencode.json fue actualizado.
